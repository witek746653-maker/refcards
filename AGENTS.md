# Refcards — правила для Codex

Эти инструкции относятся к репозиторию `refcards` и должны соблюдаться при любых изменениях кода.

## Структура проекта
- Не смешивать фронтенд и бэкенд код. Фронтенд — `frontend/`, бэкенд — `backend/`.
- Не создавать новые папки верхнего уровня без явной необходимости; по возможности добавлять файлы в существующие директории.
- Контент карточек находится в `cards/` и синхронизируется с базой данных `backend/refcards.db`.

## Контекст и стек (технические ограничения)
- Приложение лёгкое и локальное:
  - frontend: простая статическая папка `frontend/` (проект может быть SPA или статикой);
  - backend: Flask + SQLite; запуск: `python backend/server.py`.
- В деве фронтенд может быть подключён через `REFCARDS_FRONTEND` env var; по умолчанию путь `../frontend`.
- Предпочитать небольшие, безопасные изменения, не ломающие API `/api/*` и формат Markdown карточек.

## Данные карточек (файлы + БД)
- Основной источник контента — файлы в `cards/` (Markdown). БД синхронизирует метаданные и используется для быстрых запросов.
- При изменении фронт- и бэкенда сохранять совместимость импорта/экспорта Markdown.
- Сохранять front-matter поля (`name`, `domain` и т.п.) и не удалять дополнительные поля без согласования с UI.
- Импорты/синхронизация: дедуплицировать по `filepath`/`id` и не удалять карточки автоматически без явного подтверждения.

## Безопасность и деплой
- Никогда не коммитить секреты и ключи в репозиторий (`SECRET_KEY`, токены, пароли) — только через env vars.
- Для upload'ов: валидировать тип/размер, сохранять файлы в `backend/uploads/` и не доверять пользовательским путям.
- По умолчанию не менять настройки публичного хоста или порта без указания в документации.

## Автотесты и скрипты (обязательное обновление при изменениях)
- При изменении API `/api/*` или структуры Markdown обновлять соответствующие тесты и скрипты:
  - `test_my_logic.py` и другие тесты в `backend/tests/` (если добавляются) — поддерживать актуальность;
  - скрипты в `scripts/` и `tools/` при изменении форматов импорта/экспорта.

## Прочее
- Малые изменения с низким риском можно вносить напрямую, но для изменений, затрагивающих данные или API — PR с описанием миграций.
- Если предлагаете изменение формата карточек — дайте инструкции по миграции и обратной совместимости.
